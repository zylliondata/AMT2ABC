# ABC 脱敏详细指南

> 状态：正式
> 受众：维护者（审阅复核依据）、进阶贡献者
> 脱敏是 ABC 进仓库的**前置必要条件**：不脱敏，不合并。

本文是审阅 PR 时「脱敏是否合格」的判定标尺。与 [`abc-spec.md`](./abc-spec.md)、[`pr-review-sop.md`](./pr-review-sop.md) 配套。

## 1. 红线：必须清除的内容

| 找什么 | 改成 | 说明 |
|--------|------|------|
| 客户名、公司名（如「某某汽车」「XX 集团」） | `CustomerA` / `CompanyX` | 含中英文、缩写 |
| 设备型号、品牌 | `Machine` / `Device` | 含型号代号 |
| IP 地址、内网地址（`10.x` / `192.168.x` / `172.16-31.x`） | `<endpoint>` | 任何 IP 都算 |
| 域名、内网主机名 | `<endpoint>` | |
| 真实字段名（如 `dp1_temp_zk01`） | 语义化名（`temperature`） | 去除设备/工位编码痕迹 |
| 真实人名 | 删掉 / `author` / `tester` | 含拼音、工号 |
| 真实地名、厂区名 | 删掉 / 通用词 | |
| 时间戳（毫秒级，如 `1749000000000`） | `0` | 暴露产出时间 |
| 内部接口路径、服务地址 | `<endpoint>` | |

> [!warning] 红线责任
> 任何脱敏遗漏导致信息泄露，**责任由提交者承担**。维护者做最终把关，但不替贡献者背责。

## 2. `applicationInfo` 强制处理项

这一节是硬规则，**逐字段核对**，缺一不可：

| 字段 | 强制处理 | 不合格的典型 |
|------|----------|--------------|
| `id` | 必须为 `000000000000000000000000`（24 位 0） | 残留真实库 id（24 位十六进制） |
| `name` | 语义化重命名 | 残留 `test1`、`copy_of_xxx`、内部模块名 |
| `createAt` | 必须为 `0` | 残留毫秒时间戳 |
| `createBy` | 必须为 `""` | 残留创建者账号 |

## 3. `applicationDSL` 逐字段审查清单

按子节点逐一过，**重点**标 ⭐：

### inputs ⭐
- [ ] 每个入参的 `name` 语义化（非设备/工位编码）
- [ ] `defaultValue` / `testValue` 无客户、设备、IP、人名
- [ ] `description` 无业务机密

### outputs ⭐
- [ ] `name` 语义化
- [ ] `value` 表达式（`{{ ... }}`）无内部字段名泄露
- [ ] `description` 无敏感描述

### methods
- [ ] 方法名规范，无内部接口名
- [ ] `action` 内引用的组件名、方法名无内部痕迹
- [ ] `params` 无敏感参数

### event.list
- [ ] 事件名规范（`onSubmit` 这类）
- [ ] `description` 无业务机密

### global ⭐
- [ ] 变量名语义化
- [ ] 初始 `value` 无敏感内容（真实 IP、客户名等）

### ui ⭐⭐（信息最密集，最易漏）
- [ ] 标题、占位符、按钮文案无客户/设备/人名
- [ ] 组件 `name` 语义化
- [ ] `onEvent` 内的触发条件、参数无敏感值
- [ ] i18n key 引用合理

### i18n.messages ⭐
- [ ] 中英文文案均审查（常被忽略英文里的泄露）
- [ ] 删除与该 ABC 无关的冗余翻译条目

### preload ⭐⭐
- [ ] `script` 不引用内网 URL / IP / 内部服务
- [ ] `style` 无敏感资源地址

### canvasSetting / setting
- [ ] 背景图、配置无敏感信息
- [ ] `setting.description` 无敏感描述

## 4. 机器可扫的高危关键词模式

审阅时可用以下正则/关键词对整个 `abc.json` 全文扫描，命中即重点核查：

| 模式 | 正则 |
|------|------|
| IPv4 | `\b(\d{1,3}\.){3}\d{1,3}\b` |
| 内网段 | `10\.\d+\.\d+\.\d+`、`192\.168\.\d+\.\d+`、`172\.(1[6-9]\|2\d\|3[01])\.` |
| 毫秒时间戳（13 位） | `\b\d{13}\b` |
| 公司/客户标识 | `客户`、`公司`、`集团`、`Customer` |
| 设备编码风格 | `[a-z]+_?[a-z]*\d+`（如 `dp1`、`zk01`，需人工判断） |
| 内部库名 | `zyllion`、`sinocbd`、`ioap`（出现需判断是否公开署名） |

> 命中不一定等于泄露（如 `author: zylliondata` 是公开署名），需人工判断。**判断不准就打回让贡献者说明。**

## 5. 审阅结论的三种状态

| 状态 | 含义 | 处理 |
|------|------|------|
| ✅ 通过 | 脱敏干净，规范合规 | 可合并（见 pr-review-sop 风险分级） |
| ⚠️ 需整改 | 有明确泄露/不规范 | request-changes + 留整改清单 |
| ❌ 拒绝 | 严重泄露或不可救药 | close PR，与负责人讨论后处理 |
